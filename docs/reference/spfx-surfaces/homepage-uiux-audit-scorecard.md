# Homepage UI/UX Audit Scorecard

Use this scorecard with the checklist. Score each category from **0 to 4**.

Apply this scorecard only after reviewing:

- `docs/reference/ui-kit/doctrine/UI-Doctrine-SPFx-Governing-Standard.md`

If the doctrine is stricter than the scorecard, the doctrine wins.

## Scoring rubric

### 0 — Failing
- Materially non-compliant
- Structurally weak
- Cannot be accepted without major redesign

### 1 — Poor
- Directionally recognizable but materially underperforming
- Multiple critical gaps remain
- Not homepage-grade

### 2 — Adequate
- Usable but clearly below benchmark quality
- Significant refinement still required
- Acceptable only with explicit exceptions

### 3 — Strong
- Solid implementation
- Meets most acceptance expectations
- Minor gaps remain, but the application is credible and near closure

### 4 — Benchmark-grade
- Excellent
- Deliberate, polished, host-safe, and productized
- Demonstrates the class of rigor expected from flagship homepage work

## Recommended pass thresholds

### Minimum professional acceptance
- No category below **2** without an explicit written exception

### Homepage-grade acceptance
- **40+/56**
- No critical doctrine or accessibility failures
- No unresolved host-fit instability
- No obviously generic enterprise-card outcome

### Flagship / benchmark-grade acceptance
- **48+/56**
- Strong purpose-fit and persona expression
- Clear productized surface identity
- Credible compact and hosted behavior
- Evidence-backed closure
- Comparable rigor to the current hbKudos public runtime without cloning it

## Hard stop failures

A high numeric score should **not** override any of the following:

- Critical accessibility failure
- Non-compliant host behavior
- Fake-shell or duplicated SharePoint chrome
- Generic enterprise card-grid outcome as the dominant posture
- Missing loading / empty / error handling where required
- Dead CTA or broken primary interaction path
- Weak packaged result that does not match source intent
- Breakpoint behavior that technically renders but becomes stressed or low-value
- Non-use or misuse of the governed doctrine stack where it is materially expected
- Untestable or ambiguous runtime ownership seams

## Final acceptance test

Close the audit only when the application demonstrates the same class of rigor as hbKudos:

- thin-shell architecture
- governed styling
- complete enough user journey
- strong host behavior
- evidence-backed closure

Benchmark-grade does **not** mean visually identical to hbKudos. It means equivalent seriousness, quality discipline, and closure standard.

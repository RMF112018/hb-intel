## Purpose

Require hosted-runtime validation for governed SPFx surfaces so acceptance is based on actual host behavior, not source intent alone.

## Runtime validation baseline

Validation must confirm:

- host-safe shell behavior (no fake shell duplication, no host chrome competition)
- command-center/workbench composition resilience in hosted containers
- overlay/focus/keyboard viability in hosted runtime
- breakpoint and container-fit behavior in hosted conditions
- state and data-freshness cues remain credible in runtime context

## Device-depth runtime checks

Equivalent design depth must be proven for desktop, tablet, and mobile hosted contexts.

## Evidence requirements

- scored acceptance sheet aligned to doctrine model
- hosted screenshots or equivalent runtime captures
- hard-stop checklist outcome
- explicit residual risk notes, if any

## Prohibited closure posture

- source-only acceptance without hosted verification where host behavior is material;
- “looks good” closure with no scored evidence;
- numeric score closure when hard-stop failures remain unresolved.

## Pattern alignment requirement

Runtime validation must include both:

- bento/cockpit overview behavior when used;
- structured workbench behavior for workflows/forms/logs/approvals/detail review.

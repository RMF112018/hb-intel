## 3. Layout change made

### Where the spacing was applied

The spacing was applied exclusively at the **root container** — specifically the `.root` class on the top-level `<section>` element of `ProjectSitesRoot`. No child components were modified; the header, control bar, chip row, filter panel, card grid, and empty/error/shimmer states all naturally inherit the new inset because they are children of the root flex column.

### Spacing strategy

- **Mobile (default, < 768 px):** `paddingLeft / paddingRight: 16 px` (`HBC_SPACE_MD`)
- **Tablet (≥ 768 px):** `paddingLeft / paddingRight: 32 px` (`HBC_SPACE_XL`)
- **Desktop (≥ 1200 px):** `paddingLeft / paddingRight: 64 px` (`HBC_SPACE_2XL`)

Vertical padding (`paddingTop` = `HBC_SPACE_LG` = 24 px, `paddingBottom` = `HBC_SPACE_XL` = 32 px) was intentionally **not** changed — the prior vertical rhythm was already correct.

### Responsive behavior

The responsive transitions use the same `hbcMediaQuery('tablet')` and `hbcMediaQuery('desktop')` helpers that the rest of the surface already uses for its card grid, filter panel, and shimmer grid. The breakpoints therefore align with the governed `hbcBreakpoints` table (`tablet` = 768 px, `desktop` = 1200 px) — no new or ad-hoc breakpoints were introduced. The `wide` breakpoint (1600 px) was deliberately left unspecified so the webpart continues to stretch full-bleed on ultra-wide monitors; there is no arbitrary `max-width` lock that would make the content feel small on 4K displays, which matches the full-bleed product direction of the SharePoint section hosting it.

## 4. Why this approach

The container-level approach was chosen over scattered per-child spacing adjustments for five reasons:

1. **Single source of truth for the inset.** One declaration on the root ensures the header, controls, chips, card grid, empty/error container, and shimmer grid are all inset consistently. A per-child approach would have required six coordinated edits and introduced risk of drift whenever the component tree evolves.
2. **Matches the existing pattern.** The surrounding file already uses `hbcMediaQuery('tablet' | 'desktop')` for its grid breakpoints. Adding two more media-query blocks to the same pattern avoids introducing any new responsive vocabulary.
3. **Token-driven.** The three padding values (`HBC_SPACE_MD`, `HBC_SPACE_XL`, `HBC_SPACE_2XL`) come from `@hbc/ui-kit/theme` and match the values the W01r-P11 compliance closure already normalized to. No hardcoded literals, no new constants.
4. **Preserves full-bleed behavior.** Because the change is padding (not `max-width` + `margin: 0 auto`), the webpart still consumes the entire section width. Only the content is inset. This matches the `supportsFullBleed: true` manifest declaration and the prompt's explicit "no awkward fixed-width lock" guidance.
5. **Zero bundle cost.** The Vite bundle grew by only 0.15 KB (419.52 KB → 419.67 KB raw; 126.55 KB → 126.59 KB gzip), which is exactly the compiled Griffel class declarations and nothing else.

No new card-system spacing, no per-column gutter adjustments, no header-bar re-layout, and no chip-row re-layout were required — the existing inner composition already handled itself correctly once the root was inset.

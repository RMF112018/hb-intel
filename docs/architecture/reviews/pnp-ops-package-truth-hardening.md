# PnP Ops Package-Truth Hardening

## What Previous Proof Missed

The previous `hb-webparts` proof path validated mostly structural integrity:

- required archive files existed,
- component IDs appeared in package manifests,
- shim files were present and linked,
- app bundle freshness hash matched current build output.

That left a gap where package structure could pass while semantic source-to-package drift still existed, especially in per-webpart manifest fields and linkage contracts relevant to runtime behavior.

## New Proofs Added

Packaging now emits a second, machine-checkable artifact during the same run:

- `dist/sppkg/hb-webparts-package-truth-proof.json`

This proof is generated by `tools/build-spfx-package.ts` from in-memory run state plus direct `.sppkg` inspection and includes:

1. Packaging run identity and timing:
- `packagingRunId`, `generatedAt`, package file identity.

2. Source fingerprints for critical runtime inputs:
- `apps/hb-webparts/src/mount.tsx`
- PnP runtime files under `apps/hb-webparts/src/webparts/pnp/` (core runtime modules + manifest)
- SHA256, size, mtime for each file.

3. Expected vs packaged asset proof:
- expected app bundle hash from current forced build,
- expected shim mappings and hash prefixes from current shim generation,
- packaged archive asset hashes extracted from `.sppkg`.

4. Manifest linkage + semantic alignment proof per webpart:
- packaged `WebPart_<id>.xml` and embedded `ComponentManifest` extraction,
- `entryModuleId` linkage verification,
- semantic field comparison against source manifests for:
  - `id`, `alias`, `componentType`, `supportedHosts`, `preconfiguredEntries`,
  - `requiresCustomScript`, `supportsThemeVariants`, `supportsFullBleed`,
- explicit `scriptResourcePath` mapping verification to expected `shell-entry-*` asset.

5. PnP-specific runtime guard:
- verifies packaged app bundle bytes contain PnP component ID marker `9e2dd84a-a121-4fb3-a964-f43a94abf9fd`,
- verifies packaged PnP webpart linkage resolves to the expected shim entry module and path.

6. Classified pass/fail checks:
- `structuralValidity`
- `freshness`
- `sourcePackageSemanticAlignment`
- `liveRuntimeProof`

Packaging now fails fast if any required proof class fails.

## Regression Classes Now Detectable

The hardened process now detects and blocks:

- stale packaged app bundle bytes despite structural ZIP success,
- stale or wrong shim bytes/path mappings,
- source-to-packaged manifest semantic drift in required runtime fields,
- broken PnP webpart manifest linkage (`WebPart_9e2dd84a-...` missing or miswired),
- missing PnP component-ID marker in packaged runtime bundle.

## What Still Cannot Be Proven Without SharePoint Runtime Evidence

Even with stronger static package-truth checks, these remain out of scope for local packaging proof alone:

- SharePoint page-host module load order and page-level script race behavior,
- tenant/App Catalog propagation timing and CDN cache behavior after upload,
- environment-specific token/audience/runtime auth behavior on actual tenant pages,
- user/session-specific rendering outcomes.

Those still require live SharePoint runtime validation.

# Prompt 03 — Package Truth and Freshness Proofs

## Objective

Strengthen the pipeline so the generated `hb-webparts.sppkg` is not merely structurally valid, but also provably fresh and aligned with current repo source for PnP Operations.

## Prompt

```text
You are now implementing package-truth and freshness proofing for the `hb-webparts.sppkg` pipeline.

Critical files:
- `tools/build-spfx-package.ts`
- any proof/verification outputs already generated by the packaging process
- any helper scripts that can safely support stronger verification

Important working rule:
- Do not re-read files that are still within your active context or memory window unless needed for precision.

Primary problem to solve:
The current pipeline appears able to prove that a component/manifest/shell-entry exists inside the package, but not necessarily that the packaged app bundle reflects the current source state of the PnP Operations webpart.

Your job:
1. Audit the existing `.sppkg` verification logic
2. Distinguish structural verification from freshness/source-alignment verification
3. Add package-truth checks that would fail if:
   - a stale `hb-webparts-app-*.js` is packaged
   - a stale `shell-entry-*` mapping is packaged
   - a source manifest and packaged manifest drift materially
   - PnP Ops source changed but the package retained older runtime semantics
4. Introduce a durable proof artifact that can be reviewed after every build
5. Make the proof artifact specific enough that it is useful in future audits

Required behavior:
- Verify the actual packaged asset names and hashes
- Verify the packaged PnP component pathing and manifest linkage
- Verify that package truth is tied back to the current packaging run
- Verify the proof is regenerated during the packaging run, not manually after the fact

Required outputs:
1. Code changes implementing stronger proofing
2. A proof artifact produced by the packaging flow
3. A markdown summary at:
   - `docs/architecture/reviews/pnp-ops-package-truth-hardening.md`

That summary must include:
- what the previous proof missed
- what new proofs now exist
- what exact regression classes are now detectable
- what still cannot be proven without SharePoint runtime evidence

Validation required:
- produce a fresh `hb-webparts.sppkg`
- inspect it directly
- record the exact proof outputs generated by the hardened process

Favor precise, machine-checkable verification over narrative-only explanation.
```

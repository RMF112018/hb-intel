# 01 — Package vs Repo Reconciliation

## Audit approach

This reconciliation judges the attached packages against the current live implementation, not against earlier chat history.

## What the attached audit package got right

1. **Ownership boundary** — It correctly concluded that the rail is wrapper-owned and pre-shell.
2. **Preserve / change split** — It correctly recommended preserving the relationship model while redesigning the public result.
3. **Hosted proof matters** — It correctly refused to treat code-path maturity as final closure.
4. **Doctrine fit** — It correctly tied the problem to homepage doctrine rather than generic component styling alone.

## What the attached audit package underdeveloped

1. **Insufficient code specificity**
   - It did not anchor the “boxed card” diagnosis to the current CSS values in `priority-rail.module.css`.
   - It did not explicitly call out that the wrapper entry stack mirrors shell width/padding in `HbHomepageEntryStack.module.css`.

2. **Not enough issue decomposition**
   - Surface architecture, overflow interaction, wrapper envelope, breakpoint hardening, regression tests, docs refresh, and hosted closure were treated too coarsely.

3. **Weak package-level gap analysis**
   - It did not explicitly assess whether the prompt count itself was too low.
   - It did not isolate documentation and scorecard refresh as a required remediation unit.

4. **Research posture not operationalized**
   - It named premium-stack expectations, but did not convert them into specific implementation guidance such as “prefer existing `@floating-ui/react` and `@radix-ui/react-scroll-area` before adding anything new.”

## What the attached implementation prompt package got right

1. It preserved the correct non-goal: **do not move the rail into shell slot semantics**.
2. It included the required directive about not re-reading already-active files.
3. It identified the right high-level workstreams: surface rebuild, wrapper envelope, breakpoint hardening, hosted validation.

## What the attached implementation prompt package underdeveloped

1. **Prompt scope was too broad**
   - Prompt 03 combined multiple concerns that should be split into variant contract + presentation hardening + tests.
   - No dedicated prompt existed for overflow/accessibility quality.

2. **Not enough “what done looks like” detail**
   - The earlier prompts explained the intent but did not force the agent to address the exact root causes in current code.

3. **No docs/scorecard remediation prompt**
   - That omission leaves stale closure language unchallenged.

4. **Validation prompt too general**
   - It did not force a durable matrix for view mode, edit mode, zoom, tablet-width, console output, and explicit failure conditions.

## Repo drift assessment

### Confirmed correct from attached packages
- The rail remains embedded before the shell.
- The shell remains separate from rail ownership.
- Breakpoint policy and first-lane governance are real code seams, not theory.

### Directionally right but insufficiently precise
- “Rebuild the flagship surface” was right, but it should now be split into:
  1. flagship variant contract
  2. surface rebuild
  3. overflow/accessibility hardening
  4. regression proof

### Still missing from old package set
- explicit use of already-installed stack members where justified
- touch target and keyboard proof expectations
- documentation refresh requirements
- stronger hosted failure conditions

## Reconciliation verdict

The attached packages are **not wrong enough to discard their thesis**, but they are **too shallow to be the final remediation package**. They should be replaced by this expanded package, not simply edited in place.

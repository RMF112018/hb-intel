# 00 — Executive Summary

## Objective

Replace the attached audit and implementation packages with one stronger combined package that reflects live repo truth, deeper issue decomposition, better prompt specificity, and harder closure criteria.

## Bottom line

The earlier packages were **directionally correct** on the most important architectural point: the rail belongs in a wrapper-owned, pre-shell region and should not be reclassified as a shell occupant.

They were **not strong enough** in three areas:

1. they did not tie the current visual failure tightly enough to the actual live code values and seams causing it
2. they did not split the remediation into enough bounded implementation units
3. they did not define a sufficiently hard closure standard for hosted SharePoint proof, regression resistance, and documentation refresh

## Preserve vs change

### Preserve
- wrapper-owned integration seam in `hbHomepageWrapperConfig.ts`
- pre-shell render placement in `HbHomepageEntryStack.tsx`
- standalone `PriorityActionsRail` mountability
- shell container-awareness, first-lane logic, and conformance machinery
- breakpoint-governed visible-action budgeting

### Change now
- current shared rail surface language that still reads as a rounded tinted card
- overflow behavior that is acceptable but not yet flagship-grade
- wrapper envelope spacing that visually reinforces a nested-card outcome
- test / proof / scorecard rigor

## Why the hosted result still feels wrong

The current implementation compounds three conditions:

1. the wrapper entry stack mirrors shell width and padding, which makes the rail feel like a contained interior module rather than a top-band command layer
2. the shared rail surface uses a lightly tinted background, rounded border radius, and explicit border, which makes it read as a card
3. the current package set treats hosted validation as the final step but does not define a strong enough pass/fail matrix to stop weak closure claims

## What this upgraded package adds

- a sharper package-vs-repo reconciliation
- a more explicit implementation map
- research-backed guidance using the already-installed premium stack where justified
- an enhanced issue register with clearer boundaries between architecture, surface, interaction, breakpoint, and proof work
- six bounded local code-agent prompts plus a dedicated validation prompt
- explicit documentation-refresh and closure rules

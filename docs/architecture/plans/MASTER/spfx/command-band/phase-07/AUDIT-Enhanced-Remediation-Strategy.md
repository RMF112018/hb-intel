# Enhanced Remediation Strategy

## Strategic posture

This enhanced package is built around one governing rule:

**Do not disturb the architecture that is correct in order to compensate for the surface grammar that is weak.**

That means the remediation strategy is:

1. preserve the wrapper/shell/data seams
2. redesign the flagship product grammar
3. lock behavior with stronger tests
4. prove closure in hosted SharePoint

---

## What to preserve

### Wrapper ownership
Preserve:
- wrapper-owned pre-shell placement
- explicit `homepage-flagship` threading
- wrapper config extraction
- no shell-occupant migration

### Rail runtime seam
Preserve:
- list-source vs manifest-fallback behavior
- loading/empty/error resilience
- mapping seam from normalized items into shared UI contracts

### Breakpoint/presentation seam
Preserve and strengthen:
- container-aware device resolution
- presentation normalization by device class
- strategy-based overflow routing

### Existing test leverage
Preserve and expand:
- wrapper order and non-migration tests
- shared rail tests for grouped content and overflow accessibility

---

## What to rewrite

### Rewrite the audit posture
Move from:
- “the screenshot looks weak”

to:
- “these exact seams already exist, these exact seams are still weak, and these exact closure units now need to be executed”

### Rewrite the flagship problem statement
Move from:
- “the rail needs more energy”

to:
- “the live homepage path is still a row/list product model and must become a recognition-first command/launcher band while preserving host-fit boundaries”

### Rewrite closure language
Move from:
- “validate everything”

to:
- “preserve these invariants, redesign these seams, assert these behaviors, package this output, and verify these hosted conditions”

---

## What to split into separate prompt units

### Split 1 — Architecture/scope vs wrapper composition
The flagship model decision and the wrapper composition closure are related but not identical.
They should be separate prompt units.

### Split 2 — Section hierarchy vs action rendering
A section can group actions correctly and still fail as a quick-launch surface.
Those are separate prompt units.

### Split 3 — Overflow IA vs general rendering
Overflow is not a minor styling problem. It is a secondary command model with its own semantics and quality bar.

### Split 4 — Shared contracts vs visual redesign
Default/admin preview compatibility and flagship differentiation deserve their own closure unit.

### Split 5 — Testing vs hosted packaging closure
Behavioral lock-in tests and hosted SPFx proof should both be forceful, but they are not the same task.

---

## What to consolidate

### Keep accessibility, keyboard, target size, and reduced motion together
These quality concerns share the same interaction contract and should stay as one coherent unit.

### Keep final validation and hosted closure together
One final hosted closure file should force the no-deferral finish line.

## Validation strengthening

The enhanced package requires stronger closure in five categories:

### A. Architecture invariants
- wrapper-owned pre-shell placement remains intact
- no shell-occupant registration
- explicit flagship context preserved

### B. Product behavior
- flagship path visibly and structurally differs from default
- primary-vs-secondary hierarchy is obvious
- overflow feels curated, not appended

### C. Responsive/container behavior
- compact states are authored and stable
- no accidental centering or timid gutter behavior
- no horizontal-scroll dependence for primary content

### D. Accessibility/interaction
- semantic correctness for overflow pattern used
- target-size and spacing credibility
- focus visibility, focus return, reduced-motion support

### E. Hosted SPFx proof
- build/package executed
- hosted page inspected
- evidence captured
- no meaningful in-scope defects deferred

---

## Final package recommendation

Use this combined package, not the attached pair, as the new authoritative execution package for the local code agent.

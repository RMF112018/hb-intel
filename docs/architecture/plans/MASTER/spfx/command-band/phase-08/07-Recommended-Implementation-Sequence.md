# 07 — Recommended Implementation Sequence

## Guiding rule
Do not redesign blindly against the screenshot before proving hosted parity.

---

## Sequence 01 — Prove hosted parity and rebuild / redeploy as needed
### Why first
This determines whether the screenshot is showing:
- stale assets,
- stale package deployment,
- or a real code-path failure on current `main`.

### Success condition
The hosted DOM and visuals prove the current flagship path is actually active.

---

## Sequence 02 — Replace slice-then-group with flagship curation
### Why second
This is the biggest code-level root cause of sparse singleton outcomes.

### Success condition
Primary visible actions are curated for flagship utility rather than sliced linearly.

---

## Sequence 03 — Compact sparse sections and suppress redundant heading waste
### Why third
Even a better visible set can still render badly unless sparse-group rendering is corrected.

### Success condition
Singletons and near-singletons no longer waste section chrome.

---

## Sequence 04 — Promote overflow into a deliberate secondary command layer
### Why fourth
Once the primary field is curated correctly, the secondary action layer can be made more authoritative and device-appropriate.

### Success condition
Desktop/tablet overflow feels like an intentional command layer, not an appended list.

---

## Sequence 05 — Make layout normalization materially divergent by mode
### Why fifth
After parity and curation, mode differences can be hardened without masking root causes.

### Success condition
Desktop, tablet, and phone states produce visibly different, purposeful flagship outcomes.

---

## Sequence 06 — Tune density and wide-state authority
### Why sixth
This is where final field confidence, rhythm, and width authority should be tuned.

### Success condition
The band uses available space confidently without drifting back into sparse enterprise-card behavior.

---

## Sequence 07 — Add closure-grade tests and hosted proof
### Why seventh
Closure requires more than a good screenshot.
It requires:
- deterministic tests,
- packaged proof,
- hosted proof,
- and documented runtime markers.

### Success condition
The surface cannot regress silently back into the current failure mode while still “technically passing.”

---

## Sequence 08 — Resolve contract honesty gaps
### Why last
This should not block the main product-quality recovery, but it should be closed before final acceptance.

### Success condition
No misleading contract vocabulary remains.

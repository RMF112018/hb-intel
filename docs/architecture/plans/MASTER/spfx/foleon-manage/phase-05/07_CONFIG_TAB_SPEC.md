# 07 — Config Tab Spec

## Purpose

The Config tab is the admin-facing control surface for Foleon runtime readiness, list bindings, backend API readiness, token readiness, package governance, security policy, registry source, and validation proof.

In the updated registry-first model, the Config tab must not become a separate configuration silo. It should display and manage non-secret values through the `HB Platform Configuration Registry` where appropriate.

## Primary Responsibilities

The Config tab must show:

- active configuration source for each value;
- whether each value came from page/webpart override, registry, build default, or backend safe config;
- whether the value is editable in the Manager;
- whether the value is read-only runtime proof;
- whether the value is derived;
- whether the value is environment-specific;
- whether the value is a secret reference only;
- whether validation is `Valid`, `Warning`, `Blocked`, `Expired`, or `Not Validated`.

## Required Sections

### 1. Runtime Readiness

Show a readiness panel with:

```text
Can initialize
Can read content
Can write content
Can manage placements
Can sync from Foleon
Can validate SharePoint lists
Can acquire SPFx token
Backend reachable
Backend safe config reachable
```

### 2. Registry Source

Show registry health:

```text
Registry list title
Registry list GUID
Registry site URL
Registry read status
Registry write status for admins
Duplicate active key status
Secret hygiene status
Last registry validation time
```

### 3. SharePoint List Bindings

Show Foleon list IDs and source:

```text
Content Registry list GUID
Homepage Placements list GUID
Interaction Events list GUID
Sync Runs list GUID
```

Each row should display:

```text
value
source: override | registry | backend | default | missing
validation status
last validated
admin note
```

### 4. Foleon API Settings

Show only safe frontend-visible values:

```text
Foleon API base URL or backend proxy base URL
Foleon API resource / audience
Foleon docs/projects API configured status from backend safe config
Foleon OAuth configured status from backend safe config
```

Do not expose client secrets, access tokens, or raw OAuth credentials.

### 5. Origin and URL Policy

Show:

```text
Accepted Foleon origins
Allow preview URLs
Production viewer URLs only
Preview/admin review behavior
Rejected origin diagnostics
```

### 6. Package / Manifest Governance

Show:

```text
Expected Foleon manifest ID
Runtime manifest ID
Expected Foleon package version
Runtime package version
Homepage expected package version
Manifest match status
Package match status
```

### 7. Backend and Auth Readiness

Show:

```text
Function App URL
SPFx token provider readiness
API audience/resource
Backend /api/foleon/config result
Current caller authorization family, if safely available
```

Do not display bearer tokens.

### 8. Admin Diagnostics

Provide an expandable admin-only diagnostics section containing:

```text
runtime binding proof
registry resolution proof
blocked reason codes
API request correlation IDs
safe-config response shape
```

## Editable vs Read-Only Rules

Editable non-secret registry values may include:

```text
AcceptedFoleonOrigins
AllowPreview
ExpectedManifestId
ExpectedPackageVersion
HomepageExpectedPackageVersion
MarketingNewHostPageUrl
HBCentralHubSiteUrl
FoleonApiBaseUrl
FoleonApiResource
```

Read-only or derived values should include:

```text
runtime manifest ID
runtime package version
current SPFx context URL
current user proof
backend safe config booleans
last validation timestamps generated by validation script
```

Secret references only:

```text
FoleonClientSecret
FoleonClientId if treated as sensitive in tenant policy
OAuth token URL if tenant policy requires backend-only management
```

## UX Requirements

- Use clear state badges: `Live`, `Preview`, `Blocked`, `Missing`, `Registry`, `Override`, `Default`, `Secret Reference`.
- Do not make marketing users parse SharePoint list names or raw GUIDs unless they open the admin diagnostics panel.
- Provide `Validate Now` actions for admins where backend routes exist.
- Provide copyable proof snippets for deployment validation.
- Never display secret values.

## Acceptance Criteria

- Config tab clearly distinguishes registry values from page overrides.
- Missing registry values produce actionable messages.
- Secret references are visible as references only.
- Write-enabled Manager behavior is tied to registry/API/token readiness, not to hidden assumptions.

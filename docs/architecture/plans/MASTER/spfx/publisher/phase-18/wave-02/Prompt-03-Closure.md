# Phase 18 Wave 02 — Prompt 03 Closure

**Status:** Closed
**Closure date:** 2026-04-16
**Manifest bump:** `hb-publisher` `1.0.0.66` → `1.0.0.67`
**Scope:** Remove committed SPFx-toolchain generated artifacts from
version control and tighten the shell's gitignore so they cannot
return.

## What was wrong

Five SPFx gulp/webpack toolchain outputs under
`tools/spfx-shell/release/` were tracked in git from before the
ignore rules matured:

```
tools/spfx-shell/release/assets/hb-publisher-app-8e540b4e.js
tools/spfx-shell/release/assets/shell-entry-1a6f8b2c-…-3fd81f9c.js
tools/spfx-shell/release/assets/spfx-hb-publisher-d3f31daa.css
tools/spfx-shell/release/component-dependency-audit/spfx-shell.json
tools/spfx-shell/release/manifests/1a6f8b2c-….manifest.json
```

These are entirely regenerated by `tools/build-spfx-package.ts` on
every run (the Vite IIFE is copied in, `gulp bundle --ship` emits
the compiled shell asset and compiled manifests, and the orchestrator
writes the per-webpart shell-entry copies in-place). Committing them
confused audits with stale generated residue and made every
packaging run produce diffs against a historical snapshot.

`tools/spfx-shell/.gitignore` already covered `node_modules/`,
`lib/`, `dist/`, `temp/`, `sharepoint/`, and `assets/*.{js,css}`,
but **not** `release/`. The gap is why the five files had
accidentally been committed.

## What changed

### `tools/spfx-shell/.gitignore`

- Added `release/` with a comment explaining that the directory is
  the gulp/webpack toolchain output and is reproduced by the
  orchestrator on every packaging run.
- Broadened the `assets/` patterns from `*.js`, `*.css` to also
  cover `*.png`, `*.jpg`, `*.svg`, because the orchestrator stages
  the full Vite output (including image assets) into `assets/`
  before gulp's package-solution step runs.

### Intentionally NOT added to the root `.gitignore`

- A bare `release/` entry at the repo root would also hide
  `docs/architecture/release/` (7 tracked documentation files).
  Scoping the ignore to the shell's nested `.gitignore` is
  sufficient and safe.

### Untracked files

The 5 previously-committed generated artifacts were untracked via
`git rm --cached`. They remain on disk as the current packaging
run's output and are covered by the new `release/` ignore going
forward.

   - All orchestrator preflight checks passed (Node 18 resolver,

   - `git status tools/spfx-shell/release/ tools/spfx-shell/assets/`

     new ignore rules hold.

   preflight green; `hb-webparts.sppkg` 3180.3 KB unchanged; all
   four package-truth checks pass. `release/` still ignored.

   its audit line; SPFx baselines still validate.

ShellWebPart.manifest.json` is tracked source but the orchestrator

Publisher's single-manifest identity and the webparts neutral-shell
identity). This predates Prompt 03 and is out of scope here; the
orchestrator restores the Publisher state on the next Publisher
packaging run, so committing after a Publisher run yields a clean
working tree.

# 07 — Surface Remediation Requirements

## Purpose

Define required remediation by surface after shared shell/context/layout/state systems are stabilized.

## Project Home

Required outcomes:

- First scan shows project status, blockers, priority actions, and confidence.
- Shell/project context/card hierarchy are balanced.
- Procore/HBI/unified lifecycle content is not decorative; it supports decisions.
- Avoid flat KPI/card field.

## Team & Access

Required outcomes:

- Resolve any layout collapse in browser evidence.
- Access Manager, Team Viewer, and Permission Request lanes have clear hierarchy.
- Request queues and execution states are preview-safe and useful.
- Permissions and governance boundaries are explained in user language.

## Documents

Required outcomes:

- Clearly distinguish formal Project Record from working-file/document-system content.
- Explain SharePoint/OneDrive/Procore lanes as workflow paths, not generic integrations.
- Source/degraded states explain value and limitation.
- Review/permission cards are coherent and not merely placeholders.

## Project Readiness

Required outcomes:

- Blockers/readiness/constraints are ranked by operational priority.
- Readiness score/status is not isolated from next action and owner.
- Unified lifecycle content supports continuity rather than clutter.
- State confidence/source confidence is visible but not overbearing.

## Approvals

Required outcomes:

- Preview-safe approval queue/checkpoint content exists even without live data.
- Owner/status/due date/risk are clear.
- Avoid generic unavailable placeholders.

## External Systems

Required outcomes:

- Each system has specific workflow value and mapping status.
- Procore, SharePoint/OneDrive, Document Crunch, Adobe Sign, Sage/other sources are only shown where repo truth supports them.
- Deep links/launch states explain permission/source limitations.

## Control Center Settings

Required outcomes:

- Governance ownership is explicit.
- Locked/editable/preview states are distinguishable.
- Settings explain impact and owner, not just control labels.
- No administrative action appears available if it is not implemented.

## Site Health

Required outcomes:

- Security/repair/configuration/sync hierarchy is obvious.
- High-risk health issues rise above lower-priority diagnostics.
- Repair status and manual IT posture are clear.
- Procore sync/repair cards are useful and not generic.

# 02 — Shell Issue Gap Register

## Structured gap register

| ID | Shell Issue | Why the attached package treatment is insufficient | Stronger treatment required | Belongs in |
|---|---|---|---|---|
| G-01 | Shell governance registry and decision model | The attached package correctly points at governance metadata but treats it as though core registry groundwork is still absent. | Extend existing registry and decision seams into a sharper shell-owned policy model: placement classes, protected/configurable decision classes, future authorability boundaries, and explicit shell-only ownership. | Audit + Prompts |
| G-02 | Versioned persisted layout policy boundary | The attached package asks for bounded persistence but does not require a versioned, authoritative, rejectable payload contract. | Define a versioned shell layout policy object, allowed mutation set, rejection taxonomy, protected decision enforcement, and canonical examples of allowed vs rejected payloads. | Audit + Prompts |
| G-03 | Preset canonicalization and override permissions | The attached package talks about presets generically and does not force canonical semantic-role discipline or override permissions. | Audit presets and preset-derived overrides for empty-band policy, semantic-role integrity, role duplication rules, and maintainer-permitted mutation boundaries. | Audit + Prompts |
| G-04 | Shared entry-stack contract across hero, actions, and first lane | The attached package misses this issue almost entirely. | Establish a shell-owned breakpoint/entry-state contract that coordinates hero height budget, top-actions density/overflow behavior, and first-lane visibility rules across independent runtime surfaces. | Audit + Prompts |
| G-05 | Cross-surface breakpoint mismatch | The attached package never meaningfully addresses the mismatch between container-aware shell logic and viewport/device-class logic elsewhere in the entry stack. | Create a shared entry-state vocabulary and integration seam so the entry sequence is governed coherently instead of opportunistically. | Audit + Prompts |
| G-06 | Shell-owned mode negotiation | The attached package does not identify that shell-fit decisions are currently weakly surfaced to future consumers. | Formalize a shell-owned mode/conformance seam for comfort, compactness, stacking, and slot-fit outcomes so future shell consumers are not forced to infer from diagnostics or CSS data attributes alone. | Audit + Prompts |
| G-07 | Proof-grade conformance program | The attached package asks for closure artifacts but not serious proof. | Require tests, harness scenarios, preview outputs, screenshots / inspection states, and reviewable closure artifacts tied to breakpoint doctrine, persisted policy bounds, and reflow-safe behavior. | Audit + Prompts |
| G-08 | Reflow and constrained-state validation | The attached package references breakpoint doctrine but does not ask for hard conformance under constrained conditions. | Add explicit 320 CSS px / 256 CSS px and short-height conformance checks where relevant to shell-owned entry behavior and proof artifacts. | Audit + Prompts |
| G-09 | Public shell contract hardening | The attached package does not call out the risk of continuing to accept an under-typed raw shell config boundary. | Clarify the public shell config contract, version its layout policy slice, and ensure safe parsing between raw webpart properties and shell policy input. | Audit + Prompts |
| G-10 | Prompt architecture itself | The attached package uses four broad prompts, one of which is stale and none of which define closure strongly enough. | Expand to a better-sequenced prompt set that separates shell governance, persisted policy, preset canonicalization, entry-stack contract, mode negotiation, and proof. | Audit only |

## Priority view

### Highest priority
- G-04 shared entry-stack contract
- G-02 versioned persisted boundary
- G-07 proof-grade conformance program

### Medium-high priority
- G-03 preset canonicalization
- G-05 cross-surface breakpoint mismatch
- G-06 shell-owned mode negotiation

### Foundational but not the lead gap
- G-01 governance registry refinement
- G-09 public shell contract hardening

## Key judgment

The attached package over-weighted “future control panel readiness” and under-weighted “current doctrinal shell compliance.”

The stronger package must close both, but in that order:

1. define and harden the shell’s own rules
2. align the entry stack to those rules
3. prove the result

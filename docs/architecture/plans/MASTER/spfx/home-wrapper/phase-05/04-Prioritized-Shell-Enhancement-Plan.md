# 04 — Prioritized Shell Enhancement Plan

## Program goal

Close the shell-only Wave 02 scope with an implementation plan that reflects current repo truth, strengthens shell governance, and produces reviewable proof.

## Prompt architecture

### Prompt 01 — Harden Shell Governance Registry and Decision Model
**Why first**
The shell must first clarify what it owns and what it does not.

**Primary outcome**
A stronger shell policy model built on the existing registry and protected/configurable decision seams.

### Prompt 02 — Define Versioned Shell Layout Policy and Persisted Boundary
**Why second**
No future control-panel readiness is credible until the persisted input boundary is bounded and versioned.

**Primary outcome**
A policy object that can be stored, validated, previewed, and rejected safely.

### Prompt 03 — Canonicalize Presets, Band Semantics, and Override Permissions
**Why third**
Preset drift and semantic ambiguity should be eliminated before the entry-stack contract is widened.

**Primary outcome**
A cleaner preset library, clearer semantic-role rules, and explicit mutation permissions.

### Prompt 04 — Establish Entry-Stack Breakpoint Contract Across Hero, Actions, and First Lane
**Why fourth**
This is the largest shell-only doctrinal gap and should be handled as a coordinated runtime contract, not as incidental responsiveness.

**Primary outcome**
A shell-owned entry-state contract for first-view coordination.

### Prompt 05 — Add Shell-Owned Mode Negotiation and Container Conformance Seams
**Why fifth**
Once the entry stack shares a vocabulary, shell-fit outcomes should be consumable as a proper seam rather than inferred from diagnostics and CSS attributes.

**Primary outcome**
A reusable mode/conformance contract for shell behavior.

### Prompt 06 — Build Shell Conformance Harness, Tests, and Closure Artifacts
**Why last**
Proof should validate the hardened design, not guess at it.

**Primary outcome**
A reviewer-grade closure package.

## Priority judgment

### Highest execution priority
1. Prompt 04
2. Prompt 02
3. Prompt 06

### Stability priority
4. Prompt 03
5. Prompt 05
6. Prompt 01

## Recommended implementation standards across all prompts

### Use the live seam set
Do not invent a parallel shell architecture.
Extend the actual shell.

### Keep control-panel readiness bounded
Prepare policy and validation surfaces.
Do not build a maintainer UI in this scope.

### Separate shell policy from module design
Do not turn shell prompts into module work.

### Demand proof
Every significant shell decision should result in:
- tests
- preview examples
- reviewable diagnostics or artifacts

## Closure gate

The wave is not closed when the code merely compiles.

The wave is closed only when a reviewer can see that:

- the shell owns a stronger governance model
- persisted inputs are bounded
- presets are canonical
- the entry stack is contractually coordinated
- shell-fit modes are formalized
- the proof package demonstrates conformance

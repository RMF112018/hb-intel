# Ideal End-State Homepage Shell Model

## 1. Core model

The ideal homepage is a **governed entry-stack + shell system** made of:

1. **System-authored hero**
2. **Governed priority actions band**
3. **Governed first shell lane**
4. **Semantic post-hero bands**
5. **Preset-driven layout behavior**
6. **Protected rules plus controlled maintainer flexibility**

## 2. Ideal entry stack

### Hero
- independent flagship surface
- full-width
- system-authored
- height-budgeted by breakpoint
- never allowed to bury the first shell lane

### Priority actions
- compact, forceful, prioritized
- real top-task system, not directory behavior
- breakpoint-aware overflow
- explicit action budget by device class

### First shell lane
- operational/editorial proof band
- begins on first load
- major/minor composition on wide desktop
- single-column on tablet portrait and below

## 3. Ideal shell band model

### Band 1 — Operational / editorial anchor
- **major:** Project Portfolio Spotlight
- **minor:** Company Pulse

### Band 2 — Executive context
- Leadership Message

### Band 3 — Safety / field lane
- Safety Field Excellence once activated

### Band 4 — People and culture
- People & Culture Public

### Band 5 — Recognition
- HB Kudos

## 4. Ideal preset library

### Preset A — Operational flagship (recommended default)
- Spotlight major
- Pulse minor
- Leadership full
- Safety full
- People & Culture full
- Kudos full

### Preset B — Editorial emphasis
- Pulse major
- Spotlight minor
- Leadership full
- remaining bands unchanged

### Preset C — Culture-heavy campaign mode
- Leadership + People full visibility
- Spotlight remains present but demoted below first band
- used only intentionally and temporarily

## 5. Ideal footprint grammar

The shell should support more than one large-screen grammar:

- `major / minor`
- `full`
- optional future `major / minor / compact` only when validated
- explicit compact and summary-collapse modes per occupant

The shell should not expose arbitrary freeform resizing.
It should expose **approved footprints** only.

## 6. Ideal occupant rules

### System-authored
- hero remains independent
- first shell lane must begin on first load
- tablet portrait and phones remain single-column
- prohibited pairings remain prohibited
- recognition remains contextual, not dominant

### Maintainer-configurable later
- preset choice
- optional visibility of approved occupants
- limited reorder within compatible bands
- compact vs standard treatment where supported
- temporary campaign emphasis modes within approved boundaries

## 7. Ideal module fit expectations

### Company Pulse
- first-lane capable
- major or minor depending preset

### Project Portfolio Spotlight
- first-lane capable
- major by default

### Leadership Message
- secondary anchor, not default homepage opener

### Safety Field Excellence
- operational band with strong urgency semantics
- promoted when active

### People & Culture Public
- rebuilt to shared-surface/token rigor before it participates in more ambitious shell modes

### HB Kudos
- persistent recognition lane
- not first-lane dominant by default

## 8. Ideal wide-state behavior

The shell should:
- actually reach its ultrawide state
- use additional width to improve composition
- not simply widen a centered content rail
- preserve readable scan lines and controlled negative space

## 9. Ideal control-panel posture

The future control panel should behave like:
- a governed layout editor
- with presets, compatibility rules, preview, and validation

It should not behave like:
- a webpart parking-lot drag-and-drop board with arbitrary sizes

## 10. End-state definition of done

The homepage reaches the target state when:
- the entry stack behaves like one authored experience
- the first shell lane proves operational value immediately
- the strongest modules are compositionally promoted
- the weakest important module has been normalized to benchmark discipline
- wide layouts are real, not theoretical
- and future layout governance can be added without re-architecting the shell

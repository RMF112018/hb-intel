# 06 — Ideal End-State Homepage Shell Model

## End-state principle

The homepage should read as:

**brand → action → best available value**

not:

**brand → directory → empty modules → maybe value later**

## Ideal entry stack

### 1. Signature hero
- remains full-width
- keeps the current premium brand posture
- uses a tighter vertical budget on laptop baseline
- remains code-governed, not editor-freely configurable

### 2. Priority actions
- replace OOB Quick Links with `PriorityActionsRail`
- cap visible actions by breakpoint
- use `More tools` overflow
- group and rank by frequency / urgency / role relevance
- keep high-value action density without directory sprawl

### 3. First shell lane
- always begins on first load
- must contain the strongest currently available module or compatible pair
- must not allow no-data modules to consume this position

## Ideal shell lane ordering

### First lane
- Project Portfolio Spotlight (dominant)
- secondary companion only when non-empty and shell-fit-safe

### Second lane
- strongest communications signal
- likely Company Pulse or Leadership Message depending on current content strength

### Third lane
- Safety Field Excellence when urgent or materially fresh
- otherwise People & Culture or Kudos depending on freshness and momentum

### Culture / recognition zone
- keep People & Culture and HB Kudos related in narrative
- but avoid leading with them when operational and communication surfaces are stronger

## Optional surfaces

### ToolLauncherWorkHub
Valuable, but should not crowd the first screen unless it is explicitly operating as the work acceleration layer immediately under actions.

### SmartSearchWayfinding
Potentially very strong, but only if it is treated as a deliberate discovery zone rather than a generic search widget.

## End-state shell logic

The shell should evaluate:

- content availability
- freshness / strength
- slot comfort
- prohibited pairings
- breakpoint policy
- protected prominence ceilings

and then choose the best legal composition.

## Control-panel-safe end state

A future maintainer should be able to:

- choose among approved presets
- reorder only within approved compatible bands
- toggle some optional occupants
- change bounded descriptive settings

A future maintainer should **not** be able to:

- break entry-stack rules
- force portrait multi-column first lanes
- promote recognition into protected flagship positions
- pair prohibited occupants
- create arbitrary layouts

# Prioritized Remediation Plan

## Priority order

## Priority 1 — Unify measurement authority
Before expanding layouts, create one shared entry-experience measurement and breakpoint truth.

### Why first
Because all other layout and launcher decisions depend on width truth.

### Included work
- shared entry-state context/hook,
- shell + launcher adoption,
- aligned breakpoint classification,
- aligned visibility budgeting inputs.

---

## Priority 2 — Expand shell grammar safely
Introduce governed band recipes and richer slot planning.

### Why second
Because the current grammar is the main reason the homepage still looks too stack-like.

### Included work
- named band recipes,
- richer slot topology,
- recipe validation,
- prohibition rules for unsupported pairings.

---

## Priority 3 — Formalize hosted-surface fit contracts
Make zone capabilities explicit.

### Why third
Because a richer shell cannot safely exist without trustworthy occupant fit declarations.

### Included work
- shell-fit contract type additions,
- occupant registry expansion,
- nested mode declarations,
- shell resolver adoption.

---

## Priority 4 — Rebuild first-lane behavior
Make the first lane deterministic, inspectable, and vacancy-safe.

### Why fourth
Because the first lane is the most visible shell quality signal and must not degrade ambiguously.

### Included work
- ranking strategy,
- promotion rules,
- empty-state avoidance,
- explicit reasons/diagnostics.

---

## Priority 5 — Increase width confidence and density quality
Tune large-desktop behavior to stop underusing the surface.

### Why fifth
Because architecture changes need to land first, then density decisions can be tuned with confidence.

### Included work
- envelope/gutter review,
- recipe-by-width tuning,
- ultrawide lane behavior,
- safe expansion without reflow failure.

---

## Priority 6 — Add proof harness
Formalize closure evidence.

### Why sixth
Because the proof system should validate the new behavior once the critical structural changes are in place.

### Included work
- black-box assertions,
- width/zoom matrix,
- launcher/shell alignment checks,
- screenshot evidence standard.

---

## Priority 7 — Harden preset governance
Lock down the expanded shell so it stays governable.

### Why seventh
Because richer composition without stronger validation becomes unstable.

### Included work
- preset schema tightening,
- recipe allow-lists,
- protected decisions,
- config-bound enforcement.

---

## What not to do

Do not:
- start with visual-only CSS tuning,
- tweak launcher styling in isolation,
- add more presets before recipe governance exists,
- or try to “fix” the homepage by manually widening cards inside hosted modules.

That would improve symptoms while leaving the shell problem unresolved.

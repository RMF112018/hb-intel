# HB Homepage Shell — Enhanced Combined Audit and Remediation Package

## Purpose

This package is a narrowed, repo-truth, research-informed audit and remediation package for the HB Homepage shell. It is intended to replace the prior split model of a separate audit package and implementation prompt package with one combined deliverable that is easier for a local code agent to execute decisively.

## Important limitation

The execution prompt for this session explicitly referenced two prior zip packages:

- `hb-homepage-shell-audit-package.zip`
- `hb-homepage-shell-implementation-prompt-package.zip`

Those two zip files were **not present in this chat session**. Because of that, this package **cannot claim a literal file-by-file reconciliation of the missing package contents**. Instead, it does the strongest defensible version of the requested work:

1. audits the live `main` branch seams that govern the HB Homepage shell,
2. cross-checks those seams against repo doctrine and benchmark requirements,
3. uses the supplied screenshots as rendered-state evidence,
4. uses public subject-matter research to strengthen the remediation direction,
5. and produces a stronger combined audit + prompt package that a local code agent can execute now.

That limitation is documented explicitly in `01-Package-vs-Repo-Truth-Reconciliation.md`.

## What this package concludes

The main branch already contains a **serious shell architecture foundation**:
- a clear wrapper vs shell ownership contract,
- typed breakpoint policy,
- shell conformance diagnostics,
- preset-driven band composition,
- a launcher-band wrapper seam,
- and a dedicated SPFx mount path.

The current problem is **not** that the shell has no architecture.

The current problem is that the live implementation is still **too conservative, too narrow in layout grammar, too split in measurement authority, and too under-specified in hosted-surface shell-fit behavior** to reach the intended homepage standard.

In plain terms:
- the shell is smarter than a basic stack,
- but still materially behaves like a cautious stack with one paired first lane,
- while the rendered result still underuses width and still looks too restrained for the target state.

## How to use this package

1. Read `00-Executive-Summary.md`.
2. Read `03-Expanded-Findings-Register.md`.
3. Execute the prompts in the order defined by `06-Recommended-Implementation-Waves.md`.
4. Treat all waves as **in-scope now**. The waves are sequencing guidance, not permission to defer.
5. Require proof of closure for every prompt before moving on.

## Closure standard

This package should be considered complete only when the resulting implementation:
- is container-aware at the **entry experience** level, not just within isolated child seams,
- supports richer layout archetypes than a basic stack plus one fixed 3:2 pair,
- formalizes hosted-surface shell-fit contracts,
- proves no horizontal overflow or broken reflow at required host widths and zoom states,
- uses width more confidently on large desktop surfaces without harming constrained hosts,
- and produces a rendered result that is visibly and structurally stronger than the current homepage shell.

# Plan Summary

## Objective

Convert the HB Homepage from a split flagship composition into a wrapper-owned unified entry stack while preserving:
- article-mode hero reuse,
- shell ownership boundaries,
- and closure-grade responsive proof.

## Recommended sequence

1. Move hero into `HbHomepageEntryStack`.
2. Add a wrapper-owned hero config/integration seam.
3. Make hero consume shared entry-state/container truth.
4. Rebuild hero responsive behavior around explicit entry modes.
5. Close duplicate-hero hosted rollout risk.
6. Update repo-truth seams and reference composition.
7. Add diagnostics and regression proof.
8. Document hosted operational validation.

## Non-negotiable constraints

- Do not make the hero a shell occupant.
- Do not let shell config absorb hero concerns.
- Do not leave hosted cutover ambiguity for later.
- Do not sign off without proof on constrained widths and short heights.

## Definition of success

Success is a single flagship homepage runtime where:
- `HbHomepage` owns hero + launcher + shell ordering,
- the hero is responsive from shared entry-state truth,
- constrained-screen behavior is materially better,
- hosted cutover is safe,
- and repo/runtime truth is aligned.

# Ideal End-State Homepage Shell Model

## End-state principle
The homepage shell should become a **governed orchestration layer** for differentiated premium modules.

It should not become:
- a rigid static stack forever
- or a freeform drag-anything-anywhere editor

## Ideal organization

## Band 0 — Flagship top band
Keep the hero/top-band experience outside the normal configurable shell flow.

### Why
- preserves doctrine around flagship top-band integrity
- protects the most important authored surface from entropy
- avoids turning the page’s first-glance identity into an admin-controlled experiment

## Band 1 — Primary awareness band
This is the homepage’s main decision-support band.

### Recommended inhabitants
- `ProjectPortfolioSpotlight`
- `CompanyPulse`

### Behavior
- wide: governed paired layout or alternating dominant slot model
- medium: one dominant first, one demoted just below
- narrow: one fully dominant, the other summary-first or sequenced lower

### Rule
These two should never be treated as uncontrolled co-equals with arbitrary resizing.

## Band 2 — Context and leadership band
This is the “human context” band.

### Recommended inhabitants
- `LeadershipMessage`
- `PeopleCulturePublic`

### Behavior
- wide: clustered / paired if the surfaces are compatible
- medium: vertical stack with stronger shell rhythm
- narrow: leadership first, people/culture second, with more disclosure control

### Rule
People & Culture must be rebuilt or realigned so it belongs in this band at the same maturity level.

## Band 3 — Recognition band
### Recommended inhabitant
- `HbKudos`

### Behavior
- wide: prominent but not dominant
- medium: full-width persistent section
- narrow: preserve give/celebrate pathways, reduce archive density

### Rule
Recognition remains visible and re-engageable, but does not overpower the awareness bands above it.

## Band 4 — Future operational band
### Recommended inhabitant
- `SafetyFieldExcellence`
- plus future operational/intelligence modules

### Behavior
- could become a co-equal operational lane with Project Spotlight in a later phase
- should be inserted with explicit shell rules, not simply appended

## Ideal zone structure

### Zone roles
Introduce semantic slot roles such as:
- `flagship`
- `primary`
- `secondary`
- `supporting`
- `persistent`
- `contextual`
- `optional`
- `rotatable`

### Footprint classes
Introduce governed footprint options such as:
- `full`
- `wide`
- `half`
- `rail`
- `stack`
- `compact`
- `summary`

### Capability flags
Each module should declare a capability contract:
- supports full-width
- supports half-width
- supports rail placement
- minimum comfort width
- preferred height band
- supports compact mode
- supports collapse
- supports tabbed grouping
- must remain top-half or may demote lower

## Ideal adaptive behavior

### Wide
- explicit lane composition
- at least one dominant anchor
- supporting clusters rather than uniform stacking
- no accidental equal-weight dominance collisions

### Medium
- preserve a dominant first-read sequence
- collapse some pairings
- tighten density
- convert some modules to alternate shell-fit modes

### Narrow
- one major object at a time
- summary-first supporting content
- stronger progressive disclosure
- minimal dead space
- preserved CTA clarity

### Container-aware behavior
Modules should respond to their assigned slot size, not only viewport width.

That means:
- container queries where justified
- or equivalent slot-size-aware variant resolution

## Ideal motion posture
- subtle section reveal
- premium but restrained hover response
- no theatrical shell motion
- `prefers-reduced-motion` respected
- shell-level transitions only where they clarify layout or state change

## Ideal future extensibility posture
The shell should scale by:
- adding modules to a governed registry
- assigning capabilities
- assigning approved zone families
- selecting among approved layout presets

It should **not** scale by:
- hand-editing JSX order forever
- allowing unrestricted ad hoc placement
- assuming every module can become a same-size card

## End-state thesis
The ideal homepage shell is:
- authored
- adaptive
- semantically tiered
- protectively configurable
- and durable enough to grow without collapsing into clutter

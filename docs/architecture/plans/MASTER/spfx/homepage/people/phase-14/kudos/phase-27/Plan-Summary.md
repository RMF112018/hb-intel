# Plan Summary

## Objective

Close the highest-risk HB Kudos gaps in the order that maximizes doctrine compliance, architectural stability, and validation confidence.

## Recommended sequence

### Phase A — Correct doctrine-breaking seams first
1. Shared token bridge and primitive styling seam
2. Public-surface CSS doctrine closure
3. Public-surface structural composition upgrade

### Phase B — Fix weak primitive / architecture seams
4. Split overused flyout shell into purpose-specific surface shells
5. Decompose the overgrown companion runtime

### Phase C — Upgrade the companion into a stronger product-grade workspace
6. Rebuild moderation workspace ergonomics and information hierarchy
7. Harden bulk operations and operator feedback

### Phase D — Prove it
8. Expand validation to cover real UX, accessibility, and workflow behavior
9. Re-verify packaging / mount / manifest / doctrine guard invariants
10. Run final closure validation against the doctrine and the audited findings

## Why this sequence is correct

- There is no value in polishing a surface that still violates the doctrine at the token / styling layer.
- There is no value in redesigning the moderation UX while the companion file remains overgrown and hard to change safely.
- Validation must come after substantive changes, but before declaring production readiness.

## Closure standard

Every prompt in this package is a closure unit. The issue assigned to that prompt remains open until the code agent proves that:

- the full footprint was identified,
- stale and contradictory code was removed,
- the replacement is complete,
- adjacent code paths no longer preserve the same defect,
- the repo still honors the governing doctrine,
- tests and validation relevant to that finding pass.

## Preserve while remediating

The package is not asking for a blind rewrite. Preserve:
- runtime split contract,
- mount linkage,
- manifest adjacency,
- import discipline,
- typed governance actions,
- existing test guardrails that still reflect valid architecture.

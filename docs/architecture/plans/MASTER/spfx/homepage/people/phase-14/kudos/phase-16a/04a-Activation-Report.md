## Hardening applied during activation

Fixes that landed in this commit set:

- **Harness seed → React re-render coupling.** Added a
  `hb-kudos-seeded` `CustomEvent` dispatched from `__hbKudosSeed`
  and subscribed-to in `KudosTab.tsx` / `KudosCompanionTab.tsx`,
  bumping a local `seedTick` state used as a `key` on the mounted
  webpart. Without this the initial empty-store fetch left the
  hook stuck in its `listError && !listConfig` branch even after
  `invalidatePeopleCultureCache()` was called.
- **Install order fix.** `installKudosHarness()` now runs at the
  top of each tab module (not inside `useEffect`) so the
  `fetch`-replacement and `storeKudosListHostUrl` overrides are in
  place before `HbKudos` / `HbKudosCompanion` mount and trigger
  their initial data fetches.
- **List host override.** Added `storeKudosListHostUrl(HARNESS_SITE_URL)`
  alongside `storeSiteUrl` so `getKudosListHostUrl()` resolves to
  the harness URL instead of the hardcoded HBCentral, letting the
  fetch interceptor catch every read/write.
- **Empty-list handling in the interceptor.** Unknown People &
  Culture lists (announcements, celebrations) now return
  `{value: []}` for GET so `fetchPeopleCultureListData`'s parallel
  readers don't push errors that poison the top-level hook state.
- **Error / loading branches instrumented.** The `HbKudos` fallback
  branches for `isLoading` and `listError && !listConfig` now carry
  `data-hbc-testid="hb-kudos-public-root"` so the harness root is
  findable during the brief initial-fetch window.
- **Locator registry alignment.** `QueueBucket` renamed
  `'revision-requested'` → `'revisionRequested'` to match the real
  `COMPANION_TABS` id.
- **Give-trigger multiplicity.** `main-surface` spec uses
  `.first()` since four render variants of the trigger (primary,
  ghost-inline, solid, homepage) can coexist.
- **Keyboard-reachability reframed.** Rather than assume the
  preview shell's first Tab lands on Give Kudos, the spec walks
  Tab up to 40 times and asserts the Give Kudos button becomes
  the activeElement. This matches the intent ("reachable via
  keyboard") without being coupled to harness chrome tab order.

No assertion was weakened for the sake of passing.

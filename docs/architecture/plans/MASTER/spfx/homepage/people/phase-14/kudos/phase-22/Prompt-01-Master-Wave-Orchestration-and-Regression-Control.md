# Prompt 01 — Master wave orchestration and regression control

## Objective

Create and enforce a disciplined execution plan across HB Kudos Waves 1–4 so each wave builds on the last without regressing earlier gains.

## Repo source of truth

Repo:
- `https://github.com/RMF112018/hb-intel.git`

Branch:
- `main`

## Required execution posture

Treat the four real waves as follows:

### Wave 1 — doctrine/design-system foundation
Must lock:
- iconography compliance
- token discipline
- styling posture
- homepage premium-stack usage where relevant
- local surface-family governance

### Wave 2 — UI-layer architecture and shared-seam cleanup
Must build on Wave 1 and must **not** regress it.

### Wave 3 — cohesion, accessibility-system consistency, interaction clarity, host-safe hardening
Must build on Waves 1–2 and must **not** regress them.

### Wave 4 — containment, model-grade bar, validation tightening, manifest/runtime/package preservation
Must build on Waves 1–3 and must **not** regress them.

## Required implementation direction

### 1. Create a cross-wave gate model
For each wave, define:
- entry prerequisites
- in-wave objectives
- explicit regression checks against prior waves
- exit criteria before the next wave begins

### 2. Define regression categories explicitly
At minimum, later waves must be checked for regressions in:
- doctrine/import/icon/token compliance
- styling architecture discipline
- runtime decomposition / file overgrowth
- shared-seam cleanliness
- public experience cohesion
- accessibility-system treatment
- host-safe behavior
- mount / manifest / package integrity

### 3. Make the sequencing actionable
Do not write abstract guidance only.
Create concrete execution rules that a code agent can follow wave by wave.

## Constraints

- Do not invent a fifth implementation wave disguised as orchestration.
- Do not reopen earlier-wave work unless an actual regression or unresolved blocker is proven.
- Do not reduce the closure bar to “all prompts were executed.”

## Guardrails

- Do not re-read files that are already in your current context or memory unless you need fresh verification or additional surrounding context.
- Do not let one wave start before the previous wave’s exit criteria are actually met.
- Do not allow regressions to be justified as acceptable churn.

## Deliverable

Produce a concise master wave execution control note that includes:
- ordered wave sequencing
- per-wave entry/exit gates
- mandatory regression checks between waves
- explicit stop conditions if a wave introduces regressions

# Prompt-04 — HB Kudos Public Decisive Rendered Outcome Validation

```md
Objective

Validate closure based on the rendered product outcome, not on internal code quality alone.

Primary problem

Prior prompt sequences produced implementation churn without enough convincing visual change.

This prompt exists to prevent another internal-only closure claim.

Critical Constraints

- Use live repo truth.
- Do not begin by re-reading files already in active context or memory.
- Maintain strict focus on visible product outcome.
- Stay skeptical and evidence-based.
- Do not accept “better code” as equivalent to “better product.”

Hard directives

1. Validate the public homepage rendered state after Prompts 01–03.
2. Validate the open-composer state after Prompt 02.
3. Compare the current rendered state against the broken baseline screenshots.
4. If the rendered outcome is still too similar to the broken baseline, do not claim closure.
5. Make any additional focused changes necessary to achieve visible improvement before proceeding.

Required outcome

The validation pass must prove:
- the featured public spotlight is materially restored
- the composer is materially more resolved
- the overall public module visibly improved
- residual weaknesses, if any, are explicitly identified

Required deliverables

- validation notes
- before/after outcome statement
- updated screenshots
- explicit statement of whether the public product now looks materially different

Acceptance standard

This prompt is successful only if the rendered evidence clearly supports a claim of visible product improvement.
```

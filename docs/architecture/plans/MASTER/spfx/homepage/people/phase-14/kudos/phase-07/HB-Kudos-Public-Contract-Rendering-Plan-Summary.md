# HB Kudos Public Surface — Contract / Rendering Plan Summary

## Objective

Resolve the remaining narrow but critical public HB Kudos failure by restoring the featured recognition rendering path and tightening the contract between the public webpart, its public data adapters, and any shared people/culture surface dependencies.

## Current reality

The current issue is no longer a broad UI-redesign problem.

The current issue is now a **targeted contract/rendering defect**.

The key symptoms point to a likely mismatch between:

- public Kudos source data
- public view-model shaping
- shared featured-surface expectations
- fallback/render-state handling
- or layout assumptions in the rendering path

## Target production outcome

The public HB Kudos surface should finish this targeted sequence as:

- a public recognition surface whose featured spotlight renders fully and credibly
- a governed public adapter path that supplies the right contract shape to the shared surface
- a render-state implementation that fails gracefully when data is incomplete
- a stable local/shared/ui-kit boundary that is explainable and maintainable
- a freshly packaged deliverable aligned with repo truth

## Main workstreams

### Workstream 1 — Featured contract audit
Inspect the featured public-recognition contract end-to-end and identify the exact point of mismatch, omission, or silent degradation.

### Workstream 2 — Shared rendering alignment
Correct the shared rendering path only as necessary to ensure the public featured recognition can render fully and predictably without weakening governance.

### Workstream 3 — Public adapter and render-state hardening
Tighten the public webpart’s adapter logic and render-state behavior so sparse or partial data does not produce a visibly broken hero/spotlight shell.

### Workstream 4 — Targeted validation and evidence
Validate the restored public featured state and provide evidence that the product failure has actually been closed.

### Workstream 5 — Final sweep and packaging
Generate a fresh package and issue an honest targeted verdict on whether the contract/rendering defect class is closed.

## Non-goals

This package is **not** intended to reopen the broad public Kudos design direction.

It is **not** intended to redesign unrelated homepage webparts.

It is **not** intended to force UI-kit promotion where local/shared placement remains the better governed choice.

## Acceptance standard

This effort is successful only if the final public HB Kudos surface:

- renders featured recognition correctly
- no longer presents a partially empty or collapsed spotlight shell
- has a stable and governed contract/rendering path
- is backed by fresh packaging and honest evidence

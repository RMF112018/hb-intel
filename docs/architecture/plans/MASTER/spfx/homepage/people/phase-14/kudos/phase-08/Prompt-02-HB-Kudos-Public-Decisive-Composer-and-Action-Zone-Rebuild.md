# Prompt-02 — HB Kudos Public Decisive Composer and Action-Zone Rebuild

```md
Objective

Make the public HB Kudos composer feel visibly more resolved, deliberate, and production-ready.

Primary problem

The composer has improved incrementally, but it still does not read as a decisively finished public recognition flow.

This prompt is not for tiny tweaks. It is for a stronger composer/action-zone outcome.

Critical Constraints

- Use live repo truth.
- Do not begin by re-reading files already in active context or memory.
- Maintain strict focus on the public composer/action-zone experience.
- Maintain strict UI-kit and doctrine compliance.
- Design for SharePoint host coexistence, not against it.
- Do not treat small spacing changes as sufficient if the interaction still feels unresolved.

Hard directives

1. Audit the composer as a user-facing recognition flow, not just as a form.
2. Rework the composer layout, spacing, hierarchy, action zone, and footer treatment until the result is materially more deliberate.
3. If the current flyout/footer/action pattern is too generic or too weak, replace or restructure it.
4. If reusable host-aware composer/flyout behavior belongs in shared or ui-kit territory, move it there.
5. Do not finish until the open-composer screenshot looks visibly more resolved than the current state.

Required outcome

The composer must:
- feel like a productized recognition flow
- have a stable, trustworthy action zone
- coexist cleanly with SharePoint host controls
- show stronger hierarchy and completion
- look visibly more finished than the current screenshots

Required deliverables

- implementation changes
- concise explanation of the structural composer changes
- note of any new/reworked shared or ui-kit behavior
- updated open-composer screenshot evidence

Acceptance standard

This prompt is successful only if the composer/action-zone experience is materially and visibly improved, not merely adjusted.
```

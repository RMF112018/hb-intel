# HB Kudos Public Surface — Decisive Closure Prompt Package

## Purpose

This package is a **forceful, outcome-driven closure package** for the public-facing HB Kudos webpart.

It exists because the prior remediation sequences improved code quality, package freshness, and some local UI behavior, **but did not materially solve the core product failure**:

- the featured recognition area still does not render as a credible, content-carrying public spotlight
- the public surface still reads as a partially broken shell
- the composer still does not feel fully resolved in real SharePoint runtime conditions

This package is intentionally more forceful.

It is designed to stop incremental, low-impact prompt execution and instead compel the agent to make **structural, visible, outcome-oriented changes**.

## Core instruction philosophy

These prompts are written with a different posture:

- **Do not preserve a failing architecture out of caution**
- **Do not keep the current shared-surface path if it continues to produce a broken public feature**
- **Do not stop at "contract hardening" if the user-visible result is still weak**
- **Do not deliver minor refactors where a real rebuild is needed**
- **Do not claim closure without visible product improvement**

At the same time, the prompts still require:

- strict UI-kit governance
- doctrine compliance
- justified local/shared/ui-kit boundary decisions
- disciplined packaging and evidence
- honest final readiness reporting

## Execution order

Run the prompts in numeric order.

Do **not** skip to Prompt 05.

If Prompt 01 does not produce a visibly restored public featured surface, the agent should **not proceed as though the architecture is sound**. It should keep escalating the structural solution within the same prompt scope until the rendered outcome is materially different.

## What "decisive" means here

The agent is explicitly authorized — and expected — to do one or more of the following if required:

- replace the current featured-surface composition path
- stop using a shared surface path that is not fit for this public Kudos use case
- split or fork a surface family if the shared one is over-generalized and continuing to fail
- promote or create the necessary governed UI-kit/public surface pieces
- move logic back out of UI-kit if the promotion itself is the source of fragility
- restructure the public surface so the user-visible result clearly changes

The agent is **not** authorized to keep the current broken outcome simply because the existing architecture feels tidy.

## Acceptance stance

Do not accept the work until the agent can show:

- a visibly different and restored public featured recognition surface
- a materially more resolved composer/action zone
- clear explanation of any architectural escalation
- fresh packaging
- and an honest final verdict

If the outcome after these prompts still looks substantially similar to the current broken screenshots, the work should be treated as **not completed**.

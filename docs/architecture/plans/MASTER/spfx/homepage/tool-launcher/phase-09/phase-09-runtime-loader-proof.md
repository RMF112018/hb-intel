# Phase 09 — Runtime Loader Contract and SharePoint-Hosted Proof

## 1. Loader Contract Validation

### Runtime loading sequence

```
1. SharePoint page loads → SPFx framework initializes
2. SPFx AMD loader resolves manifest:
   cb7060f5-b852-4600-b912-a5f6f7221ce2.manifest.json
3. Manifest's loaderConfig.entryModuleId:
   "cb7060f5-b852-4600-b912-a5f6f7221ce2_1.0.0"
4. AMD loader loads shim:
   shell-entry-cb7060f5-b852-4600-b912-a5f6f7221ce2-540a6a2c.js
5. Shim define() registers module under the entry module ID
6. ShellWebPart.render() fires:
   a. SPComponentLoader.loadScript() loads hb-webparts-app-12dfdd01.js
   b. Bundle evaluates → globalThis.__hbIntel_hbWebparts = { mount, unmount }
   c. Shell calls __hbIntel_hbWebparts.mount(el, spfxContext, {
        webPartId: "cb7060f5-b852-4600-b912-a5f6f7221ce2",
        webPartProperties: { heading, maxGroups, maxItemsPerGroup, groups },
        assetBaseUrl: "https://..."
      })
7. mount() calls:
   a. storeSiteUrl(spfxContext.pageContext.web.absoluteUrl)
   b. WEBPART_RENDERERS["cb7060f5-..."] → createElement(ToolLauncherWorkHub, { config })
   c. createRoot(el).render(...)
8. ToolLauncherWorkHub renders:
   a. useToolLauncherData() → getSiteUrl() → fetch("Tool Launcher Contents")
   b. useResponsiveTier() → viewport width detection
   c. deriveToolLauncherPresentation(platforms, activeAudience)
   d. LauncherCompositionShell with 4 regions + overlay
```

## 2. SharePoint-Hosted Rendering Validation

### Region-by-region runtime behavior

| Region | SPFx runtime path | Fallback path (no SPFx) |
|--------|------------------|------------------------|
| **Command band** | Renders with platformCount from live list | Renders with defaults |
| **Flagship stage** | Featured platforms from `Tool Launcher Contents` list via REST API | Sample grouped config from manifest `preconfiguredEntries` |
| **Utility rail** | Notices, help, access, contacts from normalized list data | Not rendered (config fallback uses flat `HbcLauncherSurface`) |
| **Workflow shelves** | Platforms grouped by `workflowShelf` field | Not rendered (config fallback) |
| **All-platforms overlay** | Full inventory from `presentation.platformIndex` | Not rendered (config fallback) |

### SPFx context flow

```
spfxContext.pageContext.web.absoluteUrl
  → storeSiteUrl(url)
    → getSiteUrl() returns url (module-level singleton)
      → useToolLauncherData() detects siteUrl is available
        → fetchToolLauncherListItems(siteUrl)
          → REST API: /_api/web/lists/getbytitle('Tool Launcher Contents')/items
            → normalizeToolLauncherItems(rawItems)
              → LauncherPlatformRecord[]
```

### Data source fallback chain

| Condition | Data source | Rendering path |
|-----------|------------|---------------|
| SPFx context + list exists + items active | Live SharePoint list | 4-region composition shell |
| SPFx context + list exists + zero items | Live list (empty) | `HomepageEmptyState` with `listEmpty` message |
| SPFx context + list missing (404) | Fetch error → fallback | Config `preconfiguredEntries` via `HbcLauncherSurface` |
| SPFx context + network error | Fetch error → fallback | Config fallback |
| No SPFx context (local dev) | No fetch attempted | Config fallback |

## 3. Runtime Regression Audit

### Checked for runtime-only issues

| Potential issue | Status | Notes |
|----------------|--------|-------|
| `motion` import fails at runtime | ✅ No issue | `motion` is from `@hbc/ui-kit/homepage` which re-exports from `motion/react` — bundled in IIFE |
| `AnimatePresence` SSR mismatch | ✅ No issue | SPFx is client-only; no SSR |
| `useEffect` for resize listener | ✅ No issue | `useResponsiveTier()` registers resize + orientationchange listeners; cleans up on unmount |
| `useState` for overlay | ✅ No issue | React state is scoped to the component tree mounted by `createRoot` |
| `fetch` for list data | ✅ No issue | Browser's native `fetch` with `odata=nometadata` header; no polyfill needed |
| `useMemo` for searchable records | ✅ No issue | Computed once per `listPlatforms` reference change |
| Module-level cache (`_cache` in hook) | ✅ No issue | Survives across re-renders within the same page session; clears on page navigation |
| `document.addEventListener` for Escape key | ✅ No issue | Overlay registers/unregisters on mount/unmount via `useEffect` cleanup |

### No runtime regressions found

All launcher code uses standard React patterns (hooks, effects, refs, state) that work identically in the SPFx-hosted `createRoot` context and in the composition reference. No runtime-only issues identified.

## 4. Environment-Specific Findings

### What requires tenant deployment to validate

| Finding | Why it can't be validated without deployment |
|---------|---------------------------------------------|
| Live list data rendering | Requires `Tool Launcher Contents` list on HBCentral with seeded records |
| REST API auth headers | SPFx adds auth cookies/tokens automatically; not available in local dev |
| Asset CDN path resolution | `assetBaseUrl` injected by ShellWebPart from SPFx runtime context |
| Logo image loading | SVG assets must be deployed to HBCentral's asset library |
| SharePoint edit mode | Requires SP page editor context |

### What WAS validated without deployment

| Validation | Method |
|-----------|--------|
| Bundle generates correctly | Vite IIFE build + Node VM smoke test |
| .sppkg packages correctly | Full `build-spfx-package` execution |
| Manifest wiring correct | Shim proof artifact + manual audit |
| Mount dispatch correct | Code audit of UUID in `WEBPART_RENDERERS` |
| SPFx context flow correct | Code trace from `mount()` → `storeSiteUrl()` → `getSiteUrl()` → `useToolLauncherData()` |
| Fallback behavior correct | Config fallback path verified through prior phase builds |
| Interaction behavior correct | ARIA, keyboard, focus management verified through prior phase implementation |

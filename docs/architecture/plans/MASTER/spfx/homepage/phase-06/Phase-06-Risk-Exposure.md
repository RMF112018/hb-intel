# Phase 06 Risk Exposure — Homepage Operating Model and Content Governance

## Primary risk

The technical implementation is now ahead of the operating model.

That creates a common post-launch failure mode: the homepage can exist as a polished product surface while still degrading operationally because no one clearly owns freshness, rotation, review cadence, or approval boundaries.

## Major risks

### 1. Owner ambiguity
Without explicit ownership by zone and by webpart, stale or incorrect content will persist because each group will assume another group owns it.

### 2. Homepage sprawl
Once the homepage is seen as “the important page,” teams will try to push more and more content onto it. Without operating rules, it becomes cluttered and strategically weak.

### 3. Governance duplication
If homepage governance, navigation governance, and page-template governance are not clearly separated, administrators will receive conflicting direction and route requests inconsistently.

### 4. Hidden structural drift
Routine editorial changes can quietly become structural changes if no approval thresholds are defined.

### 5. Review cadence collapse
Quarterly review expectations often fail when they are documented vaguely or assigned to no one in particular.

### 6. Over-engineered process
If the operating model is too bureaucratic, people will bypass it and create informal workarounds.

## Risk posture for this phase

This phase should bias toward:

- explicit ownership
- simple approval matrices
- strong homepage singularity protection
- lightweight but real review cadence
- clear escalation rules
- low-friction day-to-day maintenance

## What would count as failure

- vague owner roles with no accountability
- no difference between “may edit” and “owns freshness”
- no stale-content handling rule
- treating homepage changes exactly like ordinary page edits
- creating governance docs that are too abstract to be followed

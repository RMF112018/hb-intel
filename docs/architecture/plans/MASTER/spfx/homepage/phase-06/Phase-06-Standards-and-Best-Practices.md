# Phase 06 Standards and Best Practices — Homepage Operating Model and Content Governance

## Governing principle

The homepage is a curated operating surface, not a generic publishing area.

Everything in this phase should reinforce that principle.

## Best practices

### 1. Assign ownership at two levels
Use both:
- **content owner** for day-to-day freshness
- **approval / governance owner** for quality and structural boundary protection

### 2. Separate routine edits from structural changes
Most homepage work should be routine:
- swap current content
- update links
- refresh messages
- remove stale items

Only a smaller class of changes should require architecture or governance review.

### 3. Use explicit freshness classes
A good policy is easier to follow when freshness is categorized clearly:
- daily
- weekly
- monthly
- event-driven
- evergreen

### 4. Prefer matrices over prose
For operators, administrators, and content owners, concise tables are more usable than long narrative explanations.

### 5. Keep the homepage singular
Do not allow the homepage package to drift into “custom webparts for any page.” That undermines the template governance established in Phase 05.

### 6. Treat stale content as a governance issue, not just an editorial issue
If something repeatedly goes stale, the problem may be ownership, workflow, or placement — not just a missed update.

### 7. Escalate only the right things
Architecture review should stay reserved for:
- boundary changes
- new homepage custom surface requests
- structural composition changes
- exceptions to homepage exclusivity

### 8. Preserve SharePoint-native operating patterns where possible
Lane C is governance-first. Avoid turning routine admin/editor work into engineering work.

## Documentation standards for this phase

New docs should:
- be written for admins and content owners, not only developers
- include role matrices and checklists
- cross-link to the existing lane-boundary, navigation, and branding docs
- clearly label homepage-specific rules versus broader SharePoint rules

## Verification best practice

For this phase, the quality bar is:
- internal consistency
- practical usability
- correct routing across docs
- preservation of already-locked architectural truths

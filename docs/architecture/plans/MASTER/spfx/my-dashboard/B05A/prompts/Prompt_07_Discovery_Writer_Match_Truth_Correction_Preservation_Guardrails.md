## Current defect to remediate

The current `discovery-repository.ts` list payload hard-writes:

```ts
MatchStatus: 'matched',
MatchConfidence: 'high',
MatchMethod: 'no-match',
```

regardless of the actual matching decision.

This is incompatible with My Projects eligibility and must be removed.

---

## Implementation scope

### 1. Persist actual matching output

Replace the forced override with:

- `input.matching.matchStatus`
- `input.matching.matchConfidence`
- `input.matching.matchMethod`

or the equivalent payload generated by the existing matching decision object.

### 2. Preserve traceability

Keep or improve:

- notes/provenance payloads;
- recordKey traceability;
- match notes.

The writer must remain auditable.

### 3. Preserve My Projects manual fields correctly

Integrate the outcomes of Prompt 06:

- when updating a row without Projects authority, do not blank operator-maintained:
  - canonical role arrays;
  - `procoreProject`;
- when matched Projects authority exists and the prompt sequence established mirror behavior, ensure writer/update behavior does not fight that contract.

If a separate post-discovery mirror/backfill lane remains the safer repo-native solution, the writer must at minimum preserve fields so that later mirror steps are not undone.

### 4. Tests

Add/update tests proving:

- matched result persists matched status;
- unmatched result persists unmatched status;
- review-required result persists review-required status;
- confidence and method are not overwritten to constants;
- legacy-only manual fields survive discovery refresh;
- matched row enrichment/preservation follows the selected repo-native design;
- previously forced override values no longer appear in writer output except where they are genuinely the matching-engine result.

# Recommended Implementation Waves

## Wave 01 — Cutover truth, legacy retirement, and doctrine cleanup
### Why first
These items clarify ownership and remove ambiguity in the active launcher boundary. They also clean up the biggest doctrinal issues without changing the overall launcher concept.

### Scope
- retire or quarantine legacy launcher family
- align version/description truth
- move dedicated package styling toward governed tokens/variants
- keep existing row / drawer / handheld contracts intact while cleaning the architecture

### Expected result
A cleaner, more authoritative launcher boundary that is easier to trust and easier to refine.

---

## Wave 02 — Accessibility hardening and closure proof
### Why second
Once the boundary and styling model are cleaner, interaction semantics and proof tooling can be finalized without chasing moving targets.

### Scope
- dialog focus lifecycle
- reduced-motion handling
- scroll-area / overflow primitive hardening
- consolidated launcher verification runner
- fresh hosted screenshot review and closure report

### Expected result
A launcher that is easier to defend as flagship-grade because accessibility and proof are no longer weak points.

## Deployment posture
Do not mix unrelated homepage-shell redesign work into these waves. The objective is launcher closure, not general homepage experimentation.

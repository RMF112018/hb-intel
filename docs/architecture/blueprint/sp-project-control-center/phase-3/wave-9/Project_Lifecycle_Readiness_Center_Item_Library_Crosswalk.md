## 4. Family Sections

### 4.1 Startup Family

Checklist-derived startup readiness obligations, contract/setup/mobilization/permits-posted controls, and preconstruction sequencing items.

### 4.2 Safety Family

### 4.3 Closeout Family

Closeout-from-day-one controls spanning document tracking, inspections, turnover, post-turnover, and PBC close-out requirements.

## 5. Required Classification Schema

Each seeded item must preserve or map the following fields:

`family`, `sourceDocument`, `sourcePage`, `sourceSection`, `sourceItemNumber`, `sourceItemText`, `sourceResponseOptions`, `normalizedTitle`, `lifecyclePhase`, `readinessDomain`, `itemType`, `criticality`, `riskTags`, `defaultOwnerRole`, `defaultReviewerRole`, `evidenceRequirement`, `downstreamIntegration`, `priorityActionTrigger`, `approvalCheckpointTrigger`, `externalSystemReference`, `notes`

## 6. Source Traceability Schema

Traceability key: `source_document + source_section + source_item_number + exact_item_text` with source page preserved from canonical library.

## 7. Ownership Classification Model

- `owned`: Wave 9 lifecycle-readiness owned item state.
- `linked`: operational detail owned by adjacent PCC module (for example Permit Log, Approvals/Checkpoints) with Wave 9 posture linkage.
- `external-reference`: external system or authority reference only; no runtime mutation implied.
- `deferred`: future implementation-specific handling, retained for traceability.

## 8. Evidence Requirement Model

Evidence semantics are requirement/link-level only. Evidence storage/ownership remains HB Document Control Center + SharePoint project record.

## 9. Priority Action Trigger Model

Priority Actions are triggered for blocked, overdue, failed-safety, missing-evidence, and external-mapping/reference failures.

## 10. Approval / Checkpoint Trigger Model

Approval/checkpoint triggers are raised for high-criticality closeout/inspection/CO gates and configured PM/PX/Safety/Accounting checkpoints.

## 11. External-System Reference Model

External references are deep-link/import-lineage posture only (for example Procore, Sage, Outlook, AHJ/Utility), unless later explicitly approved.

## 11A. Role / Action and Scoring Linkage

Crosswalk-level classification supports downstream role/action and scoring posture:

- `recommended_owner_role` aligns with Wave 9 role/action authority expectations for assignment and completion workflows.
- `evidence_required` and `evidence_type` align with evidence contract policies (`required-before-complete` and `required-before-approval`).
- `priority_action_trigger` and `approval_checkpoint_trigger` align with Priority Actions and Approvals/Checkpoints posture.
- `criticality`, `ownership_classification`, and external-reference fields align with risk-weighted readiness scoring, blocker surfacing, and gate readiness.

This section defines implementation-readiness mapping only; it does not authorize runtime model mutations.

## 12. Exact Item Text Preservation Rule

`exact_item_text` values must remain source-faithful. Normalized fields may be added but source wording must not be rewritten in this crosswalk layer.

## 13. Ambiguous Item Handling Process

Items marked “Other”, “if applicable”, or “if required” are retained verbatim and flagged in `notes_or_ambiguity` for project-specific disambiguation during implementation prompts.

## Apply-Gate Contract Summary

`ApplyGateRequest` carries:

- `environment` (`'non-production' | 'production'`),
- `targetTenant: NonProductionTargetDeclaration`,
- `manifest: ProvisioningManifest`,
- `proofArtifact: DryRunProofArtifact`,
- `proofMarkdown: string`,
- `operatorApproval: ApplyGateOperatorApproval`,
- `rollbackPosture: RollbackPosture`,
- `requestedBy`, `requestedAt`, `requestRef`,
- optional `dryRunBaselineRef` for explicit baseline-hash gating.

`ApplyGateDecision` carries:

- `decisionVersion: '0.1.0-apply-gate'`,
- `ok` and `status: 'blocked' | 'ready-for-non-production-apply' | 'not-executable-in-this-step'`,
- `errors`, `warnings`, `requiredHumanActions`,
- `validatedAt`,
- `manifestHash`, `proofArtifactHash`, `proofMarkdownHash` (always emitted, sha256 hex),
- `nonProductionOnly: true` (literal),
- `tenantMutationPerformed: false` (literal),
- `notExecutableReason` (always populated; no executor exists in this step).

## Approval State Summary

`ApplyGateOperatorApproval` is **distinct** from the proof artifact's embedded `ApprovalState` (Step 4). The proof artifact's approval state is restricted to `'not-requested' | 'pending'` with all-`null` fields and is unchanged by Step 5. The apply-gate approval supports the operator transitions:

- `not-requested` — gate returns `not-executable-in-this-step`.
- `pending` — gate returns `not-executable-in-this-step`.
- `approved-for-non-production` (with `approvalScope: 'non-production-only'`, non-empty `approvedBy` + `approvedAt`) — gate returns `ready-for-non-production-apply`.
- `rejected` — gate returns `blocked`.

Production approval is structurally invalid: `approvalScope` only accepts `'non-production-only'`, and the gate adds an error if `approved-for-non-production` is used without that scope. The validator never approves production.

## Non-Production Target Declaration Summary

`NonProductionTargetDeclaration` requires:

- `targetEnvironment: 'non-production'` literal,
- `tenantName`, `tenantIdPlaceholder` (placeholder, not a credential),
- `siteCollectionBaseUrl`, `allowedSitePathPrefix`,
- `operatorGroup`, `dataClassification`,
- `mutationMode: 'blocked-in-step-5' | 'future-non-production'`,
- optional `notes`.

No credentials, no secrets, no live tenant connection objects. Production targets are rejected.

## Rollback Posture Summary

`RollbackPosture` requires:

- `rollbackMode: 'manual-repair-plan' | 'future-automated-rollback'`,
- `manualRepairOwner` (non-empty),
- `manualRepairRunbookRef` (non-empty),
- `knownIrreversibleActions` (array; may be empty),
- `preMutationSnapshotRequired`, `postMutationValidationRequired` (booleans),
- optional `rollbackRef`.

Step 5 does not implement rollback or take snapshots; the apply gate only requires that the posture is declared up-front.

## Proof Byte-Match Policy

- **Optional in Step 5's package validator.** Callers may pass `evaluateApplyGate(request, { regeneratedProof: { json, markdown } })` to enforce byte-match, or omit it.
- **Mandatory for any future executor before tenant calls.** The future executor adapter (Step 5+) must regenerate the proof artifact pair from the manifest plus baseline metadata and verify byte-match against the supplied `proofArtifact` JSON serialization and `proofMarkdown` strings before invoking Graph or PnP. The closeout records this as a hard precondition the executor must satisfy.
- The package exports `hashJsonProofArtifact` and `hashMarkdownProofArtifact` (sha256 hex) so executors and operators can reproduce the byte-match check or commit baseline-hash refs.

## Executor Adapter Boundary

- The future executor adapter will consume an `ApplyGateRequest` whose decision evaluates to `ready-for-non-production-apply`.
- The executor must verify (in addition to whatever the apply gate already verified): manifest immutability, regenerated proof byte-match, locked mutation gate, non-production scope, rollback posture, and operator approval scope.
- The executor is the only future surface that may call Graph or PnP; `@hbc/project-site-provisioning` itself remains no-mutation.
- SPFx must not call the executor directly. Any future SPFx-originated apply request must be routed through an approved admin/control-plane flow.
- No Procore path is introduced. The package's no-secret, no-Procore-mirror, and no-tenant-mutation scans continue to gate every manifest passed to the apply gate.

## No-Execution Confirmation

- `evaluateApplyGate` is pure. No I/O. No filesystem write. No network. No tenant-aware operation.
- Every decision (blocked or ready) carries `tenantMutationPerformed: false` and `nonProductionOnly: true` as literal-typed constants.
- Every decision carries `notExecutableReason: 'Phase 2 Step 5 defines the apply-gate boundary; no executor exists in this package or any approved consumer. tenantMutationPerformed remains false.'`
- `src/apply-gate/**` files contain no `@pnp/`, `@azure/`, `@microsoft/sp-`, `spHttpClient`, or `GraphClient` reference. The existing source-import discipline scan continues to pass without exclusion expansion.
- The public-export discipline test was narrowed (it now scans only the identifier portion of each `export` line, ignoring path strings). The standalone-verb regex is unchanged; the change is purely about *what part of the export line* is scanned. Compound names like `evaluateApplyGate` and `applyGate` paths do not match `\bapply\b` in the identifier portion (case-sensitive).

## Known Limitations

- The apply-gate package validator's byte-match is opt-in. Future executors must enforce it.
- `knownIrreversibleActions` is collected as a freeform list; richer typing (e.g., enumerated action kinds) deferred.
- `rollbackPosture` does not yet declare specific snapshot mechanisms; that decision belongs to the executor implementation step.
- The apply gate does not consult any external policy engine; all rules are local to this package.
- Apply-gate decisions are not persisted to disk by this package (intentional); a future audit-trail layer may persist them.

## Intentionally Not Implemented

- Backend executor adapter, Azure Functions, PnP runner scripts, SPFx wiring, admin UI, deployment changes, CI/CD changes.
- Tenant mutations, Graph mutations, PnP calls, SharePoint creation.
- Approval-flow unlock for production. Production approval shapes are rejected by the validator.
- Procore SDK / HTTP client / secrets / mirror tables / write-back.
- New ADRs.
- SPFx solution manifest or version bumps.
- Edits under `packages/project-site-template/**`, `backend/**`, `apps/**`, `tools/**`, root workspace config.
